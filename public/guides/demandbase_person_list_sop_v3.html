Building a Person List — Example

People Tab · Global Workspace → US Workspace · TP North America
SOP
Workspace order — this matters
This is the single most common mistake. If you build a list in US Workspace first and try to use it for advertising, it won't connect. The advertising DSP reads from Global. Build there first, every time.
1
Build in Global
Create the person list in the Global workspace. Name it with the [US] prefix.
2
Name it right
Format: [US] Tier 1A — Healthcare. This naming convention keeps US assets identifiable in the Global view.
3
Mirror in US Workspace
Manually duplicate the same list in US Workspace with the [US] prefix included. This is your operational copy for reporting and orchestration.
4
Connect advertising
Ad campaigns reference the Global [US] version. Orchestration and reporting reference the US Workspace version.
No sync exists between US Workspace and Global. Every update to a person list requires a manual update in both places. When you add a new contact to [US] [Vertical] Tier 1A Leads & Contacts in US Workspace, you must also add it to the same list in Global for it to be included in active ad campaigns. Dual-maintenance is permanent.
Create the list
1
Confirm Global Workspace → Database → People → Person Lists → Create New
Name it [Vertical] Tier 1A Leads & Contacts · Type: Dynamic · Visibility: Public
2
Selector → Advanced tab — three selectors, AND logic
SELECTOR 1
Email
Operator → Exists
Use the arrow icon
SELECTOR 2
Industry Category
Operator → Equals
[Your Vertical]
SELECTOR 3
Account List
Operator → Contains
US Market
Arrow icon vs. square on Email: Arrow = SFDC contacts + Demandbase-sourced leads. Square = SFDC only. Always use the arrow — those extra contacts are net-new MQLs the moment a campaign fires against the list.
3
Set the tier in Selector 3
Type 2026 in the account list field, then select the tier. Contains US Market stays the same across all four lists — only the tier value changes per duplicate.
4
Hit the cycle button → confirm count → Save
Review the count to confirm your list criteria are capturing the right audience before saving.
Duplicate for all four tiers
Ellipsis (⋯) → Duplicate List. Swap only the account list tier value in Selector 3. Everything else stays identical.

Push to US Workspace
5
Ellipsis → Duplicate List → select US Workspace
Do this for all four tier lists. Keep the [US] prefix in both Global AND US Workspace — e.g. [US] [Vertical] Tier 1A Leads & Contacts. This keeps US assets identifiable across GEOs in campaigns, lists, account reporting, and buying group insights.
Advertising campaigns — associate Known Contacts. No exceptions.
6
When creating any ad campaign — add your person list under Known Contacts
In the campaign setup, navigate to the Audience section. Under Known Contacts, select the corresponding tier person list you just built for your vertical. This tells Demandbase to target these specific people — not just the account domain.
7
Match the person list tier to the account list tier in the same campaign
If your campaign targets [US] [Vertical] Tier 1A account list, the Known Contacts selection must be [US] [Vertical] Tier 1A Leads & Contacts. Mixing tiers defeats the prioritization model.
Without Known Contacts
Impressions fire at the account domain. Anonymous visitors. No persona match. Budget spent against whoever happens to be browsing — could be an intern, could be IT ops. No buying committee reach.
With Known Contacts
Ads deliver to your actual buying committee — the executives and decision-makers you built the list for. Person-level targeting. Every impression has a name and a role behind it.
If the Known Contacts field is blank when you launch a campaign, you are running blind. Build the list first, push to Global, then associate. This step is non-negotiable before any campaign goes live.
These lists are dynamic
As Lighthouse enriches accounts and contacts sync from SFDC, counts update automatically. Export to Remy only when pushing to a Marketo campaign — otherwise work directly off the list in Demandbase.
